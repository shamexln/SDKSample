Detailed Description

An endpoint access point and remote device validation tool.

Internally tries to resolve XAddrs and retrieve device metadata. If SSL is enabled, retrieving of metadata includes SSL handshakes to authenticate both communicating partners. Therefore, a valid DpwsEndpoint means:

  • Physical addresses of the remote device have been resolved
  • The metadata including physical addresses of hosted services have been resolved
  • If SSL is active, the remote device has successfully been authenticated.

After instantiation of DpwsEndpoint as shared pointer, use DpwsEndpoint::registerObserver() to listen to any validation activities going on.

Attention
The first validation is hold back until DpwsEndpoint::validate() is invoked.

Remarks:

  1. A re-validation can be triggered, e.g., from Hello messages or ProbeMatches messages
  2. For RESOLVE_MAX_RETRY the DpwsEndpoint uses a vector of delays that is reduced on each re-resolve until there is no delay available
  3. The reTransferGet is processed by endpoint's RetryCount (not a real transition in implementation)
  4. Retry counters and delays can be injected on construction.

virtual void S31::Sdc::Impl::Endpoint::DpwsEndpoint::validate ( )
pure virtual

Starts validation of the DpwsEndpoint.

This function can be called at any time. If DpwsEndpointObserver::invalid() or DpwsEndpointObserver::valid() was called before, the DpwsEndpoint immediately starts to (re-)validate the remote device.

Attention
Calls to this function aren't queued! If you start validating a DpwsEndpoint that is currently in between valid and invalid, the event is swallowed. Keep your own queue and start re-validation once valid or invalid is met.
Keep in mind that revalidation can be a cost-intensive activity as multiple messages are sent out to the network (some resolves, WS-TransferGet including SSL handshake). Limit the amounts of re-validations to a minimum.
