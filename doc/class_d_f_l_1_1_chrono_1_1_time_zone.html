Detailed Description

A time zone represents all the utc offset transitions for a specific geographic area.

Construction is done by providing a time-zone-string which either has the iana or the posix format (http://www.gnu.org/software/libc/manual/html_node/TZ-Variable.html). The only difference between these two formats is that the sign of the utc offsets is inverted. Thus it is impossible to detect the right format by inspecting the given string. The constructor from time zone has has been overloaded and the client has to explicitly specify the format of the tz-string. These two tz-strings construct identical time zones:

{
const TimeZone tzIana(IanaFormat("IST+02:00IDT+03:00,M03.4.4/26:00,M10.5.0/02:00"));
const TimeZone tzPosix(PosixFormat("IST-02:00IDT-03:00,M03.4.4/26:00,M10.5.0/02:00"));
ASSERT_TRUE(tzIana == tzPosix);
}
{
const TimeZone tzPosix(PosixFormat("IST+02:00IDT+03:00,M03.4.4/26:00,M10.5.0/02:00"));
const TimeZone tzIana(IanaFormat("IST+02:00IDT+03:00,M03.4.4/26:00,M10.5.0/02:00"));
TimeZone tz(IanaFormat("IST+02:00IDT+03:00,M03.4.4/26:00,M10.5.0/02:00"));
ASSERT_TRUE(tz == tzIana);
ASSERT_TRUE(tzIana != tzPosix);
tz = tzPosix;
ASSERT_TRUE(tz == tzPosix);
}
DFL::Chrono::TimeZone::TimeZone
TimeZone()
Constructs a default time zone representing UTC time.

In case of an invalid string an exception is thrown. If this is not desired take a look at DFL::Chrono::Convert::toTimeZone which returns none in case of an invalid string.

The tz-string can include the extension to POSIX.1, where the range of the hour fields in the DST rules are extended from 0..23 to -167..167, e.g. used in Israel since at least 2016.

The format supported in this parser has the following deviations from the reference format:

  • Strings starting with ':', where a file is referenced, are not supported.
  • The time zone names can have 3..10 characters (standard gives no upper limit).
  • The special case of a string "WART-4WARST,J1/0,J365/25", which switches between standard time and summer at the same date and time (e.g. Western Argentina Summer Time in prior years), should be interpreted as summer time / day light saving is effective the whole year. This is not supported, instead use a simpler string "WARST-3" which has the same effect.

IanaFormat DFL::Chrono::TimeZone::toIanaFormat ( ) const

Turns internal std/dst switching rules into a corresponding tz-string.

The returned format is semantically identical to the one the time zone has been constructed with but it is possible that is not identical on the character by character level.

void assertTzToTzString(const TimeZone& tz, const IanaFormat& expectedIana, const PosixFormat& expectedPosix)
{
const IanaFormat actualIana(tz.toIanaFormat());
const PosixFormat actualPosix(tz.toPosixFormat());
ASSERT_EQ(expectedIana , actualIana);
ASSERT_EQ(expectedPosix, actualPosix);
const TimeZone tzFromActualIana(actualIana);
const TimeZone tzFromActualPosix(actualPosix);
ASSERT_EQ(tz, tzFromActualIana);
ASSERT_EQ(tz, tzFromActualPosix);
}
// tz-string without dst-rule
assertTzToTzString(
TimeZone(IanaFormat("XST-12")),
IanaFormat("XST-12"),
PosixFormat("XST12"));
assertTzToTzString(
TimeZone(PosixFormat("XST-12")),
IanaFormat("XST12"),
PosixFormat("XST-12"));
// tz-string which does not specify default values
assertTzToTzString(
TimeZone(IanaFormat("XST-12XDT,M3.5.0,M10.5.0")),
IanaFormat("XST-12XDT,M3.5.0/2,M10.5.0/2"),
PosixFormat("XST12XDT,M3.5.0/2,M10.5.0/2"));
assertTzToTzString(
TimeZone(PosixFormat("XST-12XDT,M3.5.0,M10.5.0")),
IanaFormat("XST12XDT,M3.5.0/2,M10.5.0/2"),
PosixFormat("XST-12XDT,M3.5.0/2,M10.5.0/2"));
// tz-string which specifies default values
assertTzToTzString(
TimeZone(IanaFormat("XST-12:00XDT-11:00,M3.5.0/02:00,M10.5.0/02:00")),
IanaFormat("XST-12XDT,M3.5.0/2,M10.5.0/2"),

// fully qualified tz-string

TimeZone(PosixFormat("XST-12:45:30XDT-11:17:59,M3.5.0/22:34:56,M10.5.0/19:27:51")),

◆ toSystem()

SystemClock::TimePoint DFL::Chrono::TimeZone::toSystem ( LocalTime  lt,
Choose  choose 
) const

Returns the system time associated with given local time point in this time zone.

The problem with this conversion is, that it is not always unique. Depending on the zone and its std/dst-rules it is possible to give this function a local time which either never existed or is ambiguous because it existed twice. For this reason the caller has to specify how this function shall handle these cases. For example this time zone:

const TimeZone tzIana(IanaFormat("IST+02:00IDT+03:00,M03.4.4/26:00,M10.5.0/02:00"));
const Hours stdDstDiff(1);
const LocalTime stdEnd2020Local((Year(2020) / mar / 27).localDays() + Hours(2));
const LocalTime dstBegin2020Local((Year(2020) / mar / 27).localDays() + Hours(3));
const LocalTime dstEnd2020Local((Year(2020) / oct / 25).localDays() + Hours(2));
const LocalTime stdBegin2020Local((Year(2020) / oct / 25).localDays() + Hours(1));

Here you can see that there is a gap of one hour between the end of std-time and the beginning of dst-time. The following behaviour is implemented if a conversion of a local time during that hour is requested:

ASSERT_EQ(
SystemClock::TimePoint(dstBegin2020),
tzIana.toSystem(dstBegin2020Local - Seconds(1), Choose::Latest));
ASSERT_EQ(
SystemClock::TimePoint(dstBegin2020),
tzIana.toSystem(dstBegin2020Local - Seconds(1), Choose::Earliest));
